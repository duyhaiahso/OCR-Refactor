# Internationalization

## Requirement

The application must support two languages from the beginning:

- English
- Vietnamese

All user-facing text should be prepared for translation. Do not hard-code operational UI text directly into complex business logic.

Vietnamese text must always use proper Vietnamese diacritics. Do not write unaccented Vietnamese in UI copy, docs, validation messages, or operator-facing logs.

## Default Language Strategy

Recommended default behavior:

- default language: Vietnamese for factory/local operation
- secondary language: English for engineering, vendor support, and maintenance
- language can be switched from the authenticated app shell later

## Frontend Rules

- Keep UI labels, button text, validation text, menu names, and status messages translatable.
- Vietnamese copy must include diacritics.
- Avoid embedding long user-facing strings deep inside components.
- Prefer translation keys once the i18n package is added.
- Role and permission keys remain stable technical identifiers, but display names should be translated.

## Backend Rules

- Backend error codes should be stable and language-neutral.
- Backend can return a technical `code` and a default English `message`.
- Frontend should eventually translate known error codes for Vietnamese and English users.

Example:

```json
{
  "error": {
    "code": "AUTH_INVALID_CREDENTIALS",
    "message": "Invalid username or password"
  }
}
```

## Initial Implementation Plan

1. Keep current text simple while the UI foundation is still moving.
2. Add a frontend i18n layer before operational screens become large.
3. Move menu labels, auth labels, dashboard labels, and permission display names into locale files.
4. Add a language switcher in the app header.

## Recommended Locale Shape

```text
frontend/
  locales/
    en.json
    vi.json
```

## Important Note

This is a permanent project rule, not an optional feature. Every new user-facing screen should be built with English and Vietnamese support in mind.
